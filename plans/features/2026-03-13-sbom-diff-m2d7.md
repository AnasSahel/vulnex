# SBOM Diff

## Description

Allow users to compare two SBOMs belonging to the same product to understand what changed between versions. The diff shows added, removed, and updated components, and highlights any new CVE exposure introduced by added or updated components. This supports the CRA requirement that manufacturers track changes to their product's software composition over time.

### Current problems

1. When a user uploads a new SBOM version, there is no way to see what changed from the previous version.
2. New CVE exposure from dependency updates is invisible until manually checked.
3. Users must manually compare component lists to understand the impact of dependency updates.
4. No historical tracking of how a product's component landscape evolves.

### Target design

**Route:** `/dashboard/products/[id]/sbom-diff`

**SBOM selection:**
- Two dropdowns at the top: "Base SBOM" (older) and "Compare SBOM" (newer)
- Populated with all SBOMs for the product, labeled by filename and upload date
- Default: most recent two SBOMs pre-selected

**Diff view:**

Three sections:
1. **Added components** -- present in Compare but not in Base (matched by purl or name)
2. **Removed components** -- present in Base but not in Compare
3. **Updated components** -- present in both but with a different version

Each section shows:
- Component name, version (old -> new for updates), purl, license, supplier
- CVE count badge for the component in its respective SBOM

**New CVE exposure panel:**
- For added and updated components, show any CVEs that are newly matched
- "New CVE exposure" = CVEs linked to components in Compare SBOM that were not linked to any component in Base SBOM
- Highlighted with a warning indicator
- Each CVE shows: ID, severity, EPSS score

**Diff summary bar:**
- At the top: "N added, N removed, N updated, N new CVEs"
- Color-coded badges (green for added, red for removed, yellow for updated)

**View modes:**
- Unified view (default): single list with change indicators
- Side-by-side view: two columns showing Base and Compare

## User Stories

1. As a product owner, I want to compare SBOM versions so I can understand what dependencies changed in a release.
2. As a security analyst, I want to see new CVE exposure from dependency changes so I can assess the security impact of updates.
3. As a compliance officer, I want a record of component changes between product versions for CRA audit trails.
4. As a developer, I want to quickly see which components were added or removed so I can verify my dependency changes.

## Implementation Notes

- API route: `GET /api/products/[id]/sbom-diff?base=<sbomId>&compare=<sbomId>`
- Diff algorithm: build two maps keyed by purl (primary) or name (fallback), then compute set differences
- For "updated" detection: same purl/name exists in both maps but `version` differs
- New CVE exposure query: find CVEs in `component_cve` for Compare SBOM components that do not appear in `component_cve` for any Base SBOM component
- Compute diff server-side and return structured JSON -- do not compute in the browser
- Add `getSbomDiff(sbomIdA, sbomIdB)` query function in `app/src/lib/queries/products.ts`
- For side-by-side view, use a CSS grid with two equal columns
- Sort each section by component name for readability
- Reuse component table styling from Component Inventory UI (M2-3)
- Consider caching diff results if computation is expensive (skip for v1)
