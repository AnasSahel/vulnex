# KEV Sync Worker

## Description

Fetch the CISA KEV catalog (a single JSON file) and sync it into the `kev_entry` table.

### Current problems

1. KEV status in the dashboard is hardcoded (`kev: true/false` in mock data).
2. No way to know which CVEs are actively exploited in the wild.

### Target design

**Motia step: `sync-kev.step.ts`**

- Subscribes to: `sync.kev` topic
- API: `https://www.cisa.gov/sites/default/files/feeds/known_exploited_vulnerabilities.json`
- Strategy: full catalog download (single JSON file, ~1MB) then diff against DB

**Data mapping:**
- `kev_entry.cve_id` ← `vulnerabilities[].cveID`
- `kev_entry.vendor` ← `vulnerabilities[].vendorProject`
- `kev_entry.product` ← `vulnerabilities[].product`
- `kev_entry.vulnerability_name` ← `vulnerabilities[].vulnerabilityName`
- `kev_entry.date_added` ← `vulnerabilities[].dateAdded`
- `kev_entry.due_date` ← `vulnerabilities[].requiredAction` (parse date)
- `kev_entry.known_ransomware` ← `vulnerabilities[].knownRansomwareCampaignUse === "Known"`
- `kev_entry.notes` ← `vulnerabilities[].notes`

**Sync logic:**
- Download full catalog
- For each entry, check if a matching `cve` record exists — if not, create a minimal CVE record
- Upsert into `kev_entry`

## User Stories

1. As a user, I want to know which of my watched CVEs are in the CISA KEV catalog.
2. As the system, I need to track KEV additions for alerting.

## Priority

**Medium** — KEV is important for prioritization but the catalog is small and changes infrequently.

## Dependencies

- `2026-03-13-shared-db-package-a2k7` (database tables)
- `2026-03-13-motia-scaffold-b5m3` (Motia runtime)

## Implementation Notes

- The KEV catalog is a single JSON file — no pagination needed
- Currently ~1100 entries, grows by a few per week
- Safe to re-download the full catalog each sync (small payload)
- Match on `cve_id` for upserts
- No authentication required
