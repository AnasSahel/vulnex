# NVD Sync Worker

## Description

Fetch CVE records from the NIST National Vulnerability Database (NVD) API and store them in the `cve` and `cve_score` tables.

### Current problems

1. CVE data in the dashboard is hardcoded mock data.
2. No mechanism to ingest real CVE records from NVD.

### Target design

**Motia step: `sync-nvd.step.ts`**

- Subscribes to: `sync.nvd` topic (emitted by sync-all)
- API: `https://services.nvd.nist.gov/rest/json/cves/2.0`
- Pagination: NVD uses `startIndex` + `resultsPerPage` (max 2000)
- Incremental: uses `lastModStartDate` from `sync_log` to fetch only changes
- Rate limiting: NVD allows 5 requests per 30 seconds without an API key

**Data mapping:**
- `cve.id` ← `cve.id` (e.g., "CVE-2024-3094")
- `cve.description` ← `cve.descriptions[0].value` (English)
- `cve.published_at` ← `cve.published`
- `cve.last_modified` ← `cve.lastModified`
- `cve_score.cvss_v3_score` ← `cve.metrics.cvssMetricV31[0].cvssData.baseScore`
- `cve_score.cvss_v3_vector` ← `cve.metrics.cvssMetricV31[0].cvssData.vectorString`

**On completion:**
- Updates `sync_log` with cursor, timestamp, and item count
- Emits `nvd.synced` event with summary

## User Stories

1. As a user, I want to see real CVE data from NVD in my dashboard.
2. As the system, I need to continuously ingest new and updated CVEs.

## Implementation Notes

- Use `fetch` for HTTP requests (no extra HTTP library needed)
- NVD API returns max 2000 results per page — paginate with `startIndex`
- First sync fetches last 30 days of CVEs to seed the database
- Subsequent syncs use `lastModStartDate` for incremental updates
- Consider adding NVD API key support via env var for higher rate limits (50 req / 30s)
- Use `db.insert().onConflictDoUpdate()` for upserts
