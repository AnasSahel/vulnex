## 1) Fix `--no-cache` panic in KEV commands

### Problem
When `--no-cache` is used, `app.Cache` is not initialized, but KEV code still dereferences `c.cache`, causing a nil-pointer panic in KEV flows.

### Implementation
- Make cache usage optional in `internal/api/kev/client.go`:
  - Guard cache read with `if c.cache != nil` before `GetKEV`.
  - Guard cache write with `if c.cache != nil` before `SetKEV`.
- Keep non-cache logic unchanged:
  - If cache is unavailable, fetch from CISA feed directly.
  - Keep ETag + TTL refresh behavior only when cache exists.
- Optional: add a debug-level log indicating cache is disabled.

### Files
- `internal/api/kev/client.go`
- (Optional) `cmd/root.go` if logging/context needs adjustment.

### Acceptance Criteria
- `go run . --no-cache kev stats` executes without panic.
- `go run . --no-cache kev list` executes successfully.
- Cache-enabled behavior remains unchanged.
- Tests cover both cache=nil and cache-enabled cases.

## 2) Complete OSV lookup for CVE enrichment

### Problem
`QueryByCVE` in OSV client currently tries direct `/v1/vulns/{cve}` and otherwise returns empty results. This misses valid OSV data for CVEs that exist as aliases rather than direct OSV IDs.

### Implementation
- Extend `QueryByCVE` fallback path after direct ID miss:
  - Implement alias-capable lookup strategy using supported OSV query flow.
  - Convert returned vulnerabilities into existing `model.Advisory` and `model.AffectedPkg` using current conversion helpers.
- Preserve current direct-hit behavior.
- Return semantics:
  - API/transport failures: return `error`.
  - Valid no-match: return empty advisories/packages and `nil` error.

### Files
- `internal/api/osv/client.go`
- `internal/api/osv/types.go` (if schema fields are needed for fallback)
- `internal/api/osv/*_test.go` (new tests)

### Acceptance Criteria
- CVE alias records in OSV are returned by `QueryByCVE`.
- Direct OSV vulnerability ID lookup still works.
- True no-match returns empty results with no error.
- Tests cover direct hit, alias hit, and no-match scenarios.

## 3) Align docs and CLI help with actual behavior

### Problem
Current docs/help are inconsistent with implementation:
- README says config path is `~/.config/vulnex/config.yaml`, while code uses `~/.vulnex/config.yaml`.
- Root `--output` help text lists only a subset of supported formats.

### Implementation
- Choose one canonical config path strategy and align all references:
  - Either migrate code to XDG path (`~/.config/vulnex`) or
  - Keep `~/.vulnex` and update docs/help accordingly.
- Update root flag help for `--output` to include all supported formats.
- Audit and update references in README and config command text to avoid contradictions.

### Files
- `README.md`
- `cmd/root.go`
- `internal/config/config.go` (if path behavior changes)
- `cmd/config_cmd.go` (if displayed path/help text needs updates)

### Acceptance Criteria
- README, `config` commands, and runtime path behavior are consistent.
- `--output` help text matches all supported formatter options.
- No conflicting config-path statements remain.
